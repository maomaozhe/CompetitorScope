# 产品需求文档 (PRD)

> 多 Agent 竞品分析系统。本文档为 MVP（比赛版）的产品契约，所有设计与实施文档以此为准。
>
> 最后更新：2026-05-12

---

## 1. 产品概述

| 项 | 内容 |
|---|---|
| 产品名称 | CompetitorScope（竞品雷达）— 工作名 |
| 一句话定义 | 多 Agent 协作的竞品分析系统：自动收集、结构化分析、横向对比竞品信息，生成证据可追溯的分析报告 |
| 目标用户 | 团队通用 — PM / 技术 / 运营均可使用，不同角色通过分析维度差异化 |
| 核心价值 | 将 3-5 天的人工竞品调研压缩至 5-10 分钟，且每个结论可追溯到原始证据 |
| 产品形态 | Web 应用（Next.js 前端 + FastAPI 后端）+ CLI 辅助 |
| 竞品/参考 | Perplexity（搜索+引用）、Manus（过程透明）、Tavily Researcher |

## 2. 核心场景

### 场景 1: 快速竞品扫描（MVP 核心）

**触发**：PM 需要在下周汇报前了解某赛道的头部玩家

**流程**：
1. 用户输入一句话需求："帮我分析 AI Coding IDE 赛道的头部玩家，重点了解定价和开发者口碑"
2. 可选补充：手动添加已知竞品名/URL、选择关注维度
3. 系统返回候选竞品清单（8-10 家），用户勾选 4-5 家确认
4. 多 Agent 并发工作，左侧实时显示 agent 进度流
5. Planner 生成报告大纲，用户可调整详略
6. 报告渐进生成，用户可随时点击结论查看证据原文
7. 最终导出 Markdown 报告

**产出**：结构化 Markdown 报告 + 证据链 + 过程可回放

### 场景 2: 定向竞品深挖

**触发**：用户已知要分析哪几家，不需要系统发现竞品

**流程**：跳过步骤 3（竞品发现），直接进入抓取和分析

### 场景 3: 定期竞品追踪（V2）

> MVP 不做。赛后拓展方向：周期性运行 + 变更告警 + 趋势对比。

## 3. 用户旅程（MVP 详细版）

```
用户                          系统                             HITL 状态
─────────────────────────────────────────────────────────────────────────
1. 输入需求 ──────────────► Planner 解析需求                   
   (一句话 + 可选补充)        识别赛道 / 关注维度
                              ↓
                          Planner 搜索发现竞品
                          返回候选清单 (8-10家)
                              ↓
2. 勾选竞品确认 ◄──────── ⓵ HITL: 竞品确认                  可跳过(演示模式)
   (或跳过,全选)              ↓
                          Planner 生成报告大纲
                          (按4维度 × N竞品 拆解)
                              ↓
3. 审阅大纲 ◄────────────  ⓶ HITL: 大纲确认                  可跳过(演示模式)
   (可调详略)                  ↓
                          ┌── Collector-1 抓取竞品A ──┐
                          ├── Collector-2 抓取竞品B ──┤  并发
                          ├── Collector-3 抓取竞品C ──┤  执行
                          └── Collector-N 抓取竞品N ──┘
                              ↓
                          若某竞品抓取失败/不足
4. 补充链接/跳过 ◄──────── ⓷ HITL: 抓取失败提问              自动触发
                              ↓
                          ┌── Analyst-1 分析竞品A ───┐
                          ├── Analyst-2 分析竞品B ───┤  并发
                          └── Analyst-N 分析竞品N ───┘  执行
                              ↓
                          Comparator 横向对比
                          生成对比表 + 洞察
                              ↓
                          Writer 生成 Markdown 报告
                          (渐进输出,边写边显示)
                              ↓
5. 阅读报告 ──────────────  用户可点击结论查看证据原文
   追问某个结论 ◄─────────  ⓸ HITL: 报告追问                 随时可用
                              ↓
6. 导出 .md / 复制到飞书    完成
```

## 4. MVP 功能范围

### 4.1 做什么（In Scope）

| 模块 | 功能 |
|---|---|
| 需求输入 | 一句话需求 + 手动补充竞品/维度 |
| 竞品发现 | Planner 自动搜索赛道头部玩家 |
| 数据采集 | 公开网页/官网/文档 + 应用商店评论 + 社区声音 |
| 结构化分析 | 按 4 维模板(§5)结构化单家竞品 |
| 横向对比 | 多家对比表 + 关键洞察 |
| 报告生成 | Markdown 格式 + 内联引用 [N] |
| 证据链 | 每个结论可追溯:结论→分析依据→原始摘录→来源URL |
| 实时进度 | 左侧 agent 进度流(哪个 agent 在干什么) |
| HITL | 4 个介入点,演示模式默认全跳过 |
| 导出 | .md 下载 |

### 4.2 不做什么（Out of Scope / 非目标）

| 不做 | 原因 |
|---|---|
| 定期追踪/变更告警 | V2，需要持久化调度 |
| PPT/Slide 生成 | MVP 只出 Markdown |
| 用户注册/登录 | 比赛无需 |
| 多租户/权限 | 比赛无需 |
| 付费数据源 (Sensor Tower / SimilarWeb) | MVP 只用免费/低成本源 |
| 小红书深度爬取 | 反爬风险高,MVP 仅 Reddit/知乎/X |
| 自适应维度(agent 动态设计维度) | V2 |
| Critic agent（自审查+补强） | V2 |
| 多语言支持 | MVP 只支持中英文 |

## 6. HITL 交互设计

### 6.1 设计原则

- **可选不强制**：每个 HITL 点都可跳过
- **演示模式**：一个全局开关，开启后所有 HITL 自动跳过（默认选择最合理选项）
- **不阻塞演示**：评委看到的是顺滑流程，但你能讲"这里可以暂停让用户改"
- **超时自动推进**：30 秒无响应则自动采用默认选项

### 6.2 四个介入点

#### ⓵ 竞品确认

| 项 | 内容 |
|---|---|
| 触发时机 | Planner 完成竞品发现后 |
| 呈现形式 | 候选清单(8-10家)，每家附一行说明，勾选框 |
| 用户可做 | 勾选/取消勾选、手动添加竞品 |
| 默认行为 | 选排名前 5 家 |
| 超时 | 30s 自动确认默认 |
| LangGraph 实现 | `interrupt()` 在 Planner 节点末尾 |

#### ⓶ 大纲确认

| 项 | 内容 |
|---|---|
| 触发时机 | Planner 生成报告大纲后 |
| 呈现形式 | 大纲树(维度 × 竞品)，每节可标"详细/简略/跳过" |
| 用户可做 | 调整每节详略、添加自定义维度 |
| 默认行为 | 全部"标准" |
| 超时 | 30s 自动确认默认 |
| LangGraph 实现 | `interrupt()` 在 Planner 节点末尾（第二次 interrupt） |

#### ⓷ 抓取失败提问

| 项 | 内容 |
|---|---|
| 触发时机 | Collector 对某竞品抓取结果不足(<3 条有效源) |
| 呈现形式 | 提示"XX 竞品数据不足"，选项:补充URL/跳过该竞品/继续用已有数据 |
| 用户可做 | 贴补充链接、选择跳过或继续 |
| 默认行为 | 继续用已有数据 |
| 超时 | 15s 自动采用默认 |
| LangGraph 实现 | Collector 内部条件判断 → `interrupt()` |

#### ⓸ 报告追问

| 项 | 内容 |
|---|---|
| 触发时机 | 报告生成完毕后，用户阅读时随时 |
| 呈现形式 | 选中报告某段 → 弹出"深入调查此话题"按钮 |
| 用户可做 | 要求对某个结论/竞品进一步研究 |
| 默认行为 | 无（不自动触发） |
| LangGraph 实现 | 新建子图执行补充调研，结果追加到报告 |

## 7. Demo 剧本

### 7.1 赛道选择

**AI Coding IDE 赛道**

理由：
- 评委(技术圈)都懂，不需要解释赛道背景
- 数据源丰富(Reddit/HN/X 大量讨论)
- 竞品多且有清晰梯队，能体现 Planner 智能筛选
- 定价和口碑这两个维度信息量大，适合展示对比

### 7.2 演示输入

```
帮我分析 AI Coding IDE 赛道的头部玩家，
我想重点了解定价和开发者口碑
```

### 7.3 预期候选清单

Planner 返回：
1. Cursor
2. GitHub Copilot
3. Windsurf (Codeium)
4. Cline
5. Aider
6. Continue
7. Tabnine
8. Zed (AI features)

用户勾选：Cursor / GitHub Copilot / Windsurf / Cline / Aider（5 家）

### 7.4 演示时间线（目标 3 分钟）

| 时间 | 画面 | 讲解重点 |
|---|---|---|
| 0:00-0:15 | 输入需求 | "一句话就能启动" |
| 0:15-0:30 | Planner 返回候选清单 | "自动识别赛道头部玩家" |
| 0:30-0:45 | 用户勾选确认 | "人在环路，可控" |
| 0:45-1:30 | 左侧 agent 并发工作流 | **高光时刻**："5 个 Collector 在并行抓取，3 个 Analyst 已经开始分析" — 多 agent 可视化 |
| 1:30-2:00 | 报告渐进生成 | "Comparator 正在做横向对比" |
| 2:00-2:30 | 报告阅读 + 点击证据 | **第二高光**：点开"开发者吐槽点" → 弹出 Reddit 原帖摘录 + 来源链接 |
| 2:30-3:00 | 追问 + 导出 | "选中某段可以追问，最后一键导出" |

### 7.5 兜底策略

| 风险 | 兜底 |
|---|---|
| 比赛现场网络差 | 准备预录的 trace 回放（LangGraph checkpoint 重放） |
| LLM API 限流 | 本地缓存一套完整的 agent 运行结果 |
| 某竞品官网改版 | demo 用的 5 家竞品提前跑过一次,确认数据可达 |
| 演示超时 | 控制在 3 分钟以内,预设 timeout |

## 8. 成功标准

### 8.1 MVP 交付标准

| 指标 | 目标 |
|---|---|
| 端到端完成时间 | ≤ 5 分钟（5 家竞品 × 4 维度） |
| 报告质量 | 每个结论有 ≥1 条可追溯证据 |
| 证据准确率 | 随机抽查 5 个引用，≥80% URL 可访问且内容相符 |
| Agent 并发可视化 | 左侧进度流能实时展示 ≥3 个 agent 同时工作 |
| HITL 响应 | 每个介入点 ≤2s 响应 |
| 演示流畅度 | 3 分钟内跑完 demo,无人工干预 |

### 8.2 比赛评分预期

| 评分维度 | 我们的卖点 |
|---|---|
| 技术深度 | 多 Agent 编排 + LangGraph + 并发 fan-out + HITL |
| 产品完成度 | 端到端可用，有前后端 |
| 创新性 | 证据链可追溯 + 过程可回放 + 多 agent 可视化 |
| 演示效果 | 实时 agent 流 + 证据下钻 |

## 9. 技术约束（给架构设计的输入）

| 约束 | 说明 |
|---|---|
| Agent 数量 | 5 个：Planner / Collector / Analyst / Comparator / Writer |
| 编排框架 | LangGraph (StateGraph + Send API for fan-out) |
| Agent 通信 | 黑板模式：共享 state + 大产物落文件，agent 传引用 |
| 前端 | Next.js + SSE 订阅 agent 事件流 |
| 后端 | FastAPI（保持不变） |
| LLM | Claude Sonnet (Planner/Comparator/Writer) + Claude Haiku (Collector/Analyst) |
| 数据源 | Tavily 搜索 + httpx 抓取 + Reddit API + 应用商店评论页 |
| 存储 | SQLite (MVP) → PostgreSQL (升级) |
| HITL | LangGraph interrupt() 实现，前端通过 SSE 订阅 + REST 提交 |
